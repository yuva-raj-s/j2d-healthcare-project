Central data lake storage
Why
You need one place to accumulate data across time. ADLS Gen2 is cheap, hierarchical (like a file system), and natively integrates with both Databricks and Synapse. The medallion pattern (Raw → Bronze → Gold) gives you auditable, recoverable data at every stage.
What

How it fits
ADF writes CSVs to rawlayer → Databricks reads raw, writes validated Parquet to bronzelayer → Gold_NB writes business aggregations to goldlayer → Synapse reads goldlayer.

Azure Databricks (PySpark)
Distributed data transformation engine
Why
Transforming 1,000+ healthcare records requires joins across 3 source systems, schema validation, deduplication, and aggregation. PySpark can handle this at any scale, and Delta tables give ACID guarantees — crucial for healthcare data accuracy.
What
4 Python notebooks: Helper_NB (utilities), Raw_NB (validate → bronze), Silver_NB (model into 10 tables), Gold_NB (aggregate into 7 outputs).
How it fits
Each notebook is triggered by ADF in sequence. They all write audit records to J2D_Audit_table so every run is traceable.

Power BI doesn't natively read Parquet from ADLS. Synapse Serverless SQL acts as a virtual query engine — it reads Parquet files on the fly using OPENROWSET, exposes them as SQL views, and Power BI connects to those views via a standard SQL endpoint. No data duplication. Cost: ~$1/month.

External tables + 6 business views: Finance, Admissions, Device Utilization, Medicine Stock, Claim Status, Rejected Claims.
How it fits
Reads goldlayer Parquet via Managed Identity → Power BI connects DirectQuery to j2d-synapse-ondemand.sql.azuresynapse.net.

🥇 The Medallion Architecture

💡Why medallion? Why not just one layer?
Imagine copying patient records directly from a hospital system into a Power BI report. What happens when the doctor typo'd a date? Or a device ID was missing? Your report shows garbage. The medallion pattern adds checkpoints — each layer validates and enriches, so you always have a clean, traceable path back to the source.
LayerWhat it isFormatWho writes itQuality
RawExact copy from source systemsCSVADF Copy ActivityUnvalidated — exactly as-is
BronzeValidated, typed, null-checkedParquetRaw_NB.pySchema validated, fail-fast
SilverModelled into dim/fact tablesDeltaSilver_NB.pyCleansed, deduplicated, joined
GoldBusiness aggregationsParquetGold_NB.pyReady for BI consumption

🔗 Linked Services — the glue of ADF

A Linked Service is ADF's way of storing a connection to an external system. Think of it as a saved password + connection string. Once created, you reuse it across multiple pipelines without re-entering credentials.

A Dataset is built on top of a Linked Service — it specifies which table or file path within that connection. Linked Service = the door; Dataset = the specific room inside.

🏢
Industry note: In production, Linked Service credentials are always stored in Azure Key Vault, not hardcoded. ADF fetches the secret at runtime. This project follows that pattern.

🔑 Why Delta Tables in Silver, not Parquet?

💡Parquet vs Delta Lake
Regular Parquet files are immutable — you can't update or delete individual rows. Delta Lake adds an ACID transaction log on top of Parquet. This means if a hospital re-sends corrected records, you can run MERGE to update only the changed rows, not rewrite the entire dataset. For healthcare data (where corrections happen), this matters.

Gold layer writes plain Parquet (not Delta) because Synapse Serverless SQL reads Parquet natively with OPENROWSET. Delta format adds extra files that confuse Synapse's external table reader.

📐 Star Schema — fact and dimension tables

The Silver layer implements a star schema — the standard relational model for analytics. Dimension tables hold descriptive attributes (who, what, where). Fact tables hold measurable events (what happened, when, how much).

Example: fact_admission stores one row per hospital visit with foreign keys to dim_patient, dim_doctor, dim_bed, etc. This avoids repeating "Dr. Sharma" in every row — you store the doctor_id and join when needed.

⚡ Serverless SQL vs Dedicated SQL Pool

Cost~$5/TB scanned — ~$1/month for this project$720+/month minimum

🔐 Managed Identity — no passwords in code

Azure services can authenticate to each other using Managed Identity — an auto-managed service principal. When Synapse reads from ADLS, it uses its Managed Identity instead of a username/password. You grant the identity "Storage Blob Data Reader" role on the storage account, and Azure handles the rest. No secrets in code, no rotation needed.

AdvancedPySpark
~20 min▶
💡
Databricks runs PySpark — distributed Python that can process data across many machines. Even on a Single Node cluster (1 machine), PySpark gives you Delta Lake support, optimised Parquet reading, and a declarative SQL engine. We use Single Node + auto-terminate 15 min to minimise cost on the free trial.
🏢Why not just pandas? Pandas loads everything into memory. For 1,000 rows it would work, but the code would break at 100 million rows. PySpark is scale-agnostic — the same code runs on 1 node or 1,000 nodes. Writing it with Spark from the start is the professional choice.

Real-world context
Enterprise Standard vs. This Project
How a pure, real-world enterprise team operates daily compared to the portfolio-optimized setup in this repository.

1. Infrastructure Provisioning

🏭Enterprise Standard: Infrastructure as Code (IaC)
Real teams do not use the Azure Portal UI to click and create databases or workspaces. They write code (Terraform or Bicep) to provision infrastructure identically across Dev, Test, and Prod environments. Changes to infrastructure are reviewed via pull requests.

In this project: We use the Azure Portal directly. This is acceptable for learning and prototyping, but lacks the auditability and repeatability of IaC.

2. Secrets Management

🔐Enterprise Standard: Azure Key Vault Integration
Industry security mandates that no credentials exist in source control. Linked Services and Databricks endpoints dynamically fetch secrets via Managed Identity from an Azure Key Vault at runtime.

In this project: Passwords are listed directly inside SecureString configurations in the JSON files. While better than plaintext, true architectures never let tokens touch Git.

3. Software Engineering on Databricks

🐍Enterprise Standard: Modular Python & Unit Testing
Large organizations forbid executing monolithic code exclusively as raw Notebooks. Code is built as modular Python packages (.whl), relies on dependency managers (requirements.txt or pyproject.toml), and undergoes strict pytest unit testing before reaching standard execution layers.

In this project: We run isolated, top-down .py notebook scripts. This achieves the Medallion transformation goals but avoids strict code validation.

4. Pipeline Deployments

🚀Enterprise Standard: Databricks Asset Bundles (DABs)
Enterprise Databricks platforms utilize Databricks Asset Bundles (databricks.yml) to define workflows, testing compute resources, and artifact lifecycles natively as code, rather than running manual import-dir CLI scripts.

In this project: We deploy notebooks via the basic Databricks CLI inside CI/CD pipelines, omitting advanced bundle configurations.

5. Data Quality Enforcement

🚦Enterprise Standard: Dead-Letter Queues (DLQ)
In production, bad data cannot simply crash a pipeline or secretly pass into the bronze layer. Pipelines implement frameworks like Great Expectations or robust schema enforcement to write bad rows to a /Quarantine or Dead-Letter layer, triggering alerts for investigation.

In this project: Raw notebooks perform basic null-validation, but do not split streaming data into automated quarantine layers.

Career Growth
Interactive Interview Flashcards
Test your project knowledge! Click any question card to reveal the expert-level answer regarding specific architectural choices made during this build.
🔄
Q: Why use Databricks if ADF has Data Flows?
Orchestration vs Transformation
Answer
While ADF features "Mapping Data Flows" for visual transformations, we use Databricks (PySpark) because it offers granular control over big data logic, handles complex Delta Lake ACID transactions with ease, and is generally more cost-effective for massive-scale parallel processing. Here, ADF acts strictly as the orchestrator (the conductor) while Databricks is the compute engine (the musicians).
🎖️
Q: Defend the Medallion Architecture.
Why three layers?
Answer
The Medallion pattern (Raw → Bronze → Silver → Gold) guarantees data traceability and recovery. If the Silver transformation logic has an error, we don't need to re-query the original healthcare source databases (which impacts their operational load); we simple reprocess the data sitting immutably in the Raw/Bronze Data Lake. It physically separates messy extraction from clean reporting.
🏛️
Q: Why Serverless Synapse?
Serverless vs Dedicated Pool
Answer
We chose Serverless SQL Pools because it costs $5 per Terabyte queried and allows us to read Gold layer Parquet files directly from ADLS Gen2 using OPENROWSET without duplicating the data. Dedicated SQL Pools cost over $700/month idle and require loading data directly into the database storage engine. For reporting layers without massive concurrency needs, Serverless is infinitely more optimal.
🔑
Q: Why Delta format in Silver, but Parquet in Gold?
Storage format incompatibilities
Answer
Delta Lake (built on Parquet) provides ACID guarantees. In the Silver layer, we need to run MERGE statements to update incorrect hospital records without rewriting entire partitions. However, Synapse Serverless SQL connects easily to raw Parquet files, while the accompanying Delta transaction log artifacts sometimes restrict Synapse parsing. Consequently, Gold aggregated views are written out as raw Parquet strings for seamless Power BI consumption.
🚦
Q: How would you handle dirty data?
Dead-Letter Queues
Answer
In an Enterprise setup, I would enforce schema bounds in the Raw_NB.py notebook. Instead of letting bad records (e.g. string text in an integer blood_pressure field) crash the PySpark job, I would invoke a Dead-Letter Queue (DLQ) pattern. Correct rows pass into the Bronze layer, while corrupt rows are partitioned out to a /Quarantine folder in ADLS Gen2. A Logic App would then alert the engineering team of the quarantined file.
🔐
Q: How is security handled?
RBAC and Managed Identities
Answer
No passwords sit in code. The pipeline relies entirely on Role-Based Access Control (RBAC) and Managed Identities. Synapse and Databricks are given the "Storage Blob Data Reader" role to query ADLS. Any external tokens needed (like Databricks deployment tokens) are passed from Azure Key Vault implicitly, ensuring that developers only interact with connection references rather than sensitive strings.
